# ADR 0004 — 메시지 큐 도입 여부

## 맥락

비동기 수집·청킹·색인 파이프라인은 작업 큐, 재시도, 중복 방지, 상태 조회가
필요하다(FR-5·FR-6). 차터는 정당화할 수 없는 구성요소를 제거하라고 명시한다.

## 결정

초기에는 전용 브로커 없이 DB 기반 작업 테이블 + 워커 폴링으로 구현한다.
RabbitMQ는 처리량·팬아웃 요구가 DB 큐를 넘어설 때 도입하는 업그레이드 경로로
문서화한다.

## 근거

- 목표 규모에서 재시도·중복 방지·상태 관리는 DB 작업 테이블로 충족 가능하다.
- 전용 브로커는 운영 비용을 더하므로 명확한 필요가 확인될 때 도입한다.

## 대안

- RabbitMQ 즉시 도입: 이력서상 RabbitMQ 노출 가치가 있으나, 차터의 "정당화
  불가 구성요소 제거" 원칙과 충돌해 초기 기본값에서 제외하고 업그레이드 경로로 남김.
- Kafka: 로그 스트리밍급 규모가 아니라 과함으로 기각.

## 결과

- M3는 DB 작업 테이블로 단순하게 시작한다.
- 도입 트리거(처리량 임계·팬아웃 필요)를 명시해 둔다.
- 이 결정은 잠정이며 M3 실측 처리량으로 재검토한다.
