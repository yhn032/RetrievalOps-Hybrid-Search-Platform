# 협업·개발 워크플로

이 문서는 사람(사용자)과 에이전트(Claude·Codex)가 이 프로젝트를 함께 개발하는
방식을 정의한다. 세션이나 에이전트가 바뀌어도 이 기준이 계속 적용된다.
[PROJECT.md](../../PROJECT.md)와 [WORKFLOW.md](../../WORKFLOW.md)에서 이 문서를 참조한다.

## 역할 경계

| 주체 | 담당 | 비담당 |
|---|---|---|
| 에이전트 | 프레임워크·스캐폴드 구성, 설계·WBS 작성, 코드 리뷰·피드백, 문서·게이트 유지 | 제품 코드·테스트 코드 작성 |
| 사용자 | 모든 제품 코드와 테스트 코드 구현, 설계·피드백 반영, 최종 결정 | — |

핵심 원칙: **에이전트는 골격(구조·인터페이스·계약·규약)만 잡고, 구현과 테스트는
전적으로 사용자가 한다.** 에이전트는 사용자가 작성한 코드의 품질을 리뷰로 끌어올린다.

## 반복 피드백 루프

```text
1. 에이전트: 대상 기능/컨테이너의 프레임워크 제시
   (디렉터리·인터페이스·계약·환경변수·규약, 빈 골격까지만)
2. 사용자: 골격 위에 제품 코드와 테스트 코드 구현
3. 에이전트: 구현 리뷰 → 피드백(아래 기준)
4. 사용자: 수정
5. 기준 충족까지 3↔4 반복
```

WBS 항목 하나를 한 번의 루프 단위로 삼는다. 한 항목이 기준을 충족하면 다음 항목으로
넘어간다.

## 리뷰 기준

에이전트는 다음 관점으로 피드백한다. 우선순위는 위에서 아래로.

1. 정확성·버그: 명백한 결함, 경계 조건, 오류 처리
2. 설계 적합성: ADR·모듈 경계·인터페이스 계약 준수
3. 테스트: 커버리지, 의미 있는 단정, 실패 케이스 포함 (테스트 코드는 사용자가 작성)
4. 간결성: 과한 추상화·중복 제거 (Karpathy 규칙)
5. 성능: 핫패스·쿼리·색인 경로의 명백한 비효율
6. 보안: 비밀값 미포함, 입력 검증
7. 컨벤션: 언어·프로젝트 규약, 로깅·설정 방식

피드백은 "무엇을·왜·어떻게"를 함께 제시하되 **수정 코드를 대신 작성하지 않는다.**
필요하면 의사코드나 방향만 보여 준다.

## 게이트와 검증

- 문서·메타데이터·링크는 기존 게이트(`scripts/meta/*`)로 검증한다.
- 컨테이너 런타임 규칙은 `runtime-rules-check.sh`로 검증한다.
- 단위·통합 테스트의 작성과 실행은 사용자가 담당하며, 에이전트는 결과를 리뷰한다.

## 진행 추적

작업 분해와 진행 상태는 [WBS](wbs.md)로 추적한다. 설계 기준은
[시스템 설계](system-design.md)를 따른다.
